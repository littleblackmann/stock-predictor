## 完成項目

### 1. 預測記錄「正確」欄位永遠顯示「—」（v1.5.1）

#### 問題
4/8 執行預測後，預測記錄的「實際」「漲跌%」「正確」三欄全部顯示「—」，按「更新實際結果」也無法回填。

#### 根因（三層）

**第一層：exe 寫入未落盤**

PyInstaller 打包的 exe 環境中，`csv.DictWriter` 寫入 `prediction_log.csv` 後沒有呼叫 `flush()` / `os.fsync()`。資料可能只停留在 OS buffer，程式關閉時被丟棄。

證據：prediction_log.csv 最後修改日期停在 4/3（上次從原始碼環境操作的日期），4/8 的 exe 預測從未真正寫入磁碟。

**第二層：yfinance 回傳 NaN**

即使記錄存在，backfill 用 yfinance 抓 4/9 收盤價時，0050.TW、2884.TW、6116.TW 等台股的 OHLC 全部回傳 NaN（Volume 正常）。`_near_price()` 找不到有效價格，回填靜默失敗。

**第三層：背景執行緒覆蓋資料**

`_save_all()` 採「讀取→修改→全量覆蓋」模式。如果背景回填執行緒在主執行緒 `append()` 之後才執行 `_save_all()`，新追加的記錄會被覆蓋。

#### 修復（4 處）

| 位置 | 修復 |
|------|------|
| `append()` | 寫入後加 `f.flush()` + `os.fsync(f.fileno())`，並記錄 log |
| `backfill_actuals()` | yfinance 加 `repair=True`；找不到價格時標記 `actual_return = "資料延遲"` |
| `_save_all()` | 寫入前重讀 CSV，合併期間可能被 append 的新記錄 |
| `_near_price()` | 已有 `math.isnan()` 過濾，搭配 repair 後更可靠 |

**影響範圍**：`data/prediction_logger.py`

### 3. 即時行情顯示 nan（v1.5.2）

#### 問題
右上角即時行情顯示 `6116.TW nan nan(nan%)`。

#### 根因
`get_latest_price()` 用 `ticker.history(period="5d")` 抓最近 5 天資料，最新一天的 Close 是 NaN。直接拿 NaN 去算 `change` 和 `change_pct`，全部變成 nan。

#### 修復

```python
# 加 repair=True
hist = ticker.history(period="5d", auto_adjust=True, repair=True)
# 過濾殘餘 NaN
hist = hist.dropna(subset=["Close"])
```

**影響範圍**：`data/yfinance_adapter.py` → `get_latest_price()`

### 4. K 線圖缺少當日 K 棒（v1.5.2）

#### 問題
4/9 執行預測，K 線圖只顯示到 4/8，缺少 4/9 的 K 棒。

#### 根因
`fetch_history()` 下載的 4/9 資料 OHLC 全是 NaN（Volume 正常），被 `_clean_data()` 的 `dropna()` 移除。

注意：這跟 v1.4.1 修過的「K 線日期缺當天」是**不同問題**。v1.4.1 是 yfinance `end` 參數不含當天（差 1 天）；這次是資料有抓到但內容是 NaN。

#### 修復

```python
# fetch_history 加 repair=True
df = ticker.history(
    start=start_date.strftime("%Y-%m-%d"),
    end=end_date.strftime("%Y-%m-%d"),
    auto_adjust=True,
    repair=True,  # 修復 NaN / 幣值混淆
)
# TypeError fallback 也加 repair=True
df = ticker.history(period=period_str, auto_adjust=True, repair=True)
```

**影響範圍**：`data/yfinance_adapter.py` → `fetch_history()`

## 技術筆記

### yfinance `repair=True` 的運作原理

Yahoo Finance 偶爾會回傳 NaN 或異常值的收盤價（常見於台股、港股）。`repair=True` 會：
1. 偵測 NaN 或明顯異常的 OHLC 值
2. 從 Yahoo 的其他 API 端點（如 `chart` API）交叉比對
3. 嘗試修復幣值混淆（例如日圓 100 倍誤差）
4. 無法修復的仍然回傳 NaN（但大部分情況能救回來）

### 為什麼「放著不動反而壞掉」？

這次三個 bug 的共同根因都是 **yfinance 上游資料品質波動**：
- Yahoo Finance API 對台股的資料品質不穩定，某些日期的 OHLC 會突然全部回傳 NaN
- 之前剛好沒遇到 NaN 的日子，程式碼中沒做防禦
- `repair=True` 是 yfinance 內建的自癒機制，之前沒啟用

教訓：**依賴外部 API 的程式，即使程式碼不動，資料品質變化也會產生新 bug。防禦性程式設計（repair、NaN 過濾、fallback）不是可選的。**
