### 情緒權重壓縮調整

**問題：** Brave 深度分析上線後，極端情緒分數（如 -0.72）搭配動態權重（23.4%），導致 ML 預測被下修 16.8 個百分點（52.8% → 35.9%），情緒過度主導預測結果。

**根本原因：**
1. 新聞媒體標題天生偏聳動，GPT 給出的情緒分數容易極端化
2. 動態權重上限 25% 太高，high impact × short timeframe 讓權重飆到 23.4%
3. 沒有對極端分數做任何緩衝

**修復（兩處調整）：**

**1. 情緒分數壓縮函數（tanh 緩衝）**
- 公式：`compressed_score = 0.5 × tanh(raw_score)`
- 效果：極端分數自動衰減，避免媒體聳動標題過度影響
  - ±0.3 → ±0.15（輕微影響，幾乎減半）
  - ±0.5 → ±0.23
  - ±0.7 → ±0.30
  - ±1.0 → ±0.38（封頂）

**2. 動態權重上限降低（25% → 15%）**
- `min(base_weight, 0.25)` → `min(base_weight, 0.15)`
- 即使 high impact + short timeframe，情緒權重最多 15%

**調整前後對比（0050.TW 實例）：**

| | 調整前 | 調整後 |
|---|---|---|
| 原始情緒分數 | -0.72 | -0.72 |
| 壓縮後分數 | -0.72（無壓縮） | -0.42 |
| 動態權重 | 23.4%（上限25%） | 15.0%（上限15%） |
| 調整量 | -16.8% | -6.3% |
| ML 原始 | 52.8% | 52.8% |
| 最終機率 | 35.9% | 46.5% |

**設計理念：** 情緒分析應該是 ML 模型的「參考意見」，不是「推翻者」。壓縮+限重確保新聞情緒能適度修正預測方向，但不會推翻技術面和籌碼面的基本判斷。

### 模型自我進化機制

**目標：** 讓預測系統「越用越準」— 每次重訓都基於上一代模型繼續學習，而非從零開始。

**目前瓶頸分析：**
1. 訓練資料固定 700 天滑動窗口 → 資料量不增長
2. 每次重訓 = 從頭訓練 → 不保留歷史學習成果
3. 單一模型預測 → 容易過擬合特定時期的走勢
4. 52% 準確率門檻太低 → 幾乎等於猜硬幣
5. 所有特徵等權參與 → 雜訊特徵拉低預測品質

---

- `auto_retrain_worker.py`：同步修改
- 效果：訓練資料從 ~2.8 年擴大到 ~6 年

#### 2. Ensemble 投票機制（3 模型 TimeSeriesSplit）

**改動：** `lgbm_classifier.py` 大幅重寫

**原理：**
- 不再訓練 1 個模型，改用 `TimeSeriesSplit(n_splits=3)` 產生 3 組訓練/測試集
- 每組各訓練一個 LightGBM 模型
- 預測時 3 個模型**各自獨立預測**，取**平均機率**作為最終結果

**TimeSeriesSplit 示意（假設 1000 筆資料）：**
```
Fold 1: 訓練 [0:250]   → 測試 [250:500]
Fold 2: 訓練 [0:500]   → 測試 [500:750]
Fold 3: 訓練 [0:750]   → 測試 [750:1000]
```
- 每個 fold 看到不同長度的歷史 → 捕捉不同時期的市場模式
- 平均化可降低單一模型的過擬合風險

**Log 輸出範例：**
```
Ensemble Fold 1/3 | 準確率：0.5423 | F1：0.7012
Ensemble Fold 2/3 | 準確率：0.5612 | F1：0.7189
Ensemble Fold 3/3 | 準確率：0.5734 | F1：0.7301
Ensemble 訓練完成 | 平均準確率：0.5590 | 平均 F1：0.7167
預測結果：上漲 54.2%（M1=53.1% | M2=54.8% | M3=54.7%）
```

**eval_metrics 擴充：**
```json
{
  "accuracy": 0.5590,
  "f1_score": 0.7167,
  "ensemble_size": 3,
  "fold_details": [
    {"fold": 1, "accuracy": 0.5423, "f1": 0.7012},
    {"fold": 2, "accuracy": 0.5612, "f1": 0.7189},
    {"fold": 3, "accuracy": 0.5734, "f1": 0.7301}
  ],
  "incremental": false
}
```

#### 3. 增量式訓練（init_model）

**原理：** LightGBM 的 `fit(init_model=...)` 參數允許從已有模型繼續訓練，而非從零開始。

**機制：**
- 每次重訓前，自動載入上一代的 Ensemble 模型
- 每個 fold 用對應的上一代模型作為 `init_model`
- 新模型在舊模型的基礎上繼續添加決策樹 → 保留歷史學習成果
- Log 會標示「增量學習」或「全新訓練」

**效果：**
- 第 1 次訓練：全新訓練（無上一代）
- 第 2 次重訓：基於第 1 代繼續訓練（增量）
- 第 N 次重訓：基於第 N-1 代繼續訓練（增量）
- 模型會累積越來越多的市場經驗，而非每次都忘記前一代學到的東西

---

#### 4. 特徵自動篩選（min_gain_to_split）

**改動：** LightGBM 參數新增 `min_gain_to_split: 0.01`

**原理：**
- LightGBM 在每次分裂決策樹節點時，會計算分裂帶來的「信息增益」
- `min_gain_to_split = 0.01` 表示：若某次分裂的增益 < 0.01，**自動跳過**
- 效果等同自動忽略雜訊特徵（增益太低的特徵不會被用來分裂）
- 不需要手動選擇或移除特徵，LightGBM 自己判斷哪些特徵有用

---

#### 5. 準確率門檻提高（52% → 55%）

- `data/prediction_logger.py`：`accuracy_threshold: 0.52` → `0.55`
- 效果：自動重訓的標準更嚴格
- 準確率低於 55% 才會觸發自動重訓，逼模型持續提升
- 搭配增量式訓練，每次重訓都在更高的基線上繼續進步

#### 1. 美股隔夜訊號（+2~4% 預估提升）

**原理：** 美股凌晨 5 點收盤 → 台股 9 點開盤，昨晚美股表現幾乎一定影響台股當日走勢。

**實作：**
- 使用 yfinance 下載 S&P 500（`^GSPC`）、費半指數（`^SOX`）、VIX 恐慌指數（`^VIX`）
- 日期對齊：US 日期 + 1 個營業日（`pd.offsets.BDay(1)`）→ 對齊台股隔日
- 美股有時區資訊（`America/New_York`），台股沒有 → 用 `tz_localize(None)` 統一去除

**新增特徵（4 維）：**

| 特徵 | 說明 | 給 LSTM | 給 LightGBM |
|------|------|---------|-------------|
| `sp500_return` | S&P 500 隔夜對數報酬率 | ✅ | ✅ |
| `sox_return` | 費半指數隔夜對數報酬率 | ✅ | ✅ |
| `vix_level` | VIX 恐慌指數水位（÷100 標準化） | ❌ | ✅ |
| `vix_change` | VIX 日變化率 | ❌ | ✅ |

**降級策略：** 美股資料下載失敗時填入預設值（0 / 0.2），`min_gain_to_split` 自動忽略無用特徵。

#### 2. 信心篩選機制（+3~5% 預估提升，量化基金核心策略）

**原理：** 不是讓模型變準，而是讓模型「不確定的時候閉嘴」。只在有把握時預測，自然提高命中率。

**雙重判斷標準：**
- **距離 50% 幅度**（conviction）：預測結果離 50% 越遠，模型越有信心
- **Ensemble 分散度**（agreement）：3 個模型預測越一致，結果越可靠

| 信心等級 | 條件 | UI 顯示 |
|---------|------|---------|
| 🔴 low | 距離 < 3% 或 分散度 > 5% | ⚠️ 模型信心不足，建議觀望 |
| 🟡 medium | 距離 < 8% 或 分散度 > 3% | 🔶 模型信心中等 |
| 🟢 high | 距離 ≥ 8% 且 分散度 ≤ 3% | ✅ 模型高度一致 |

**UI 實作：** 預測面板新增 `label_confidence` 標籤，有顏色邊框，位於漲跌進度條下方。

#### 3. 多時間框架分析（+1~2% 預估提升）

**原理：** 日線看的是短期波動，但真正的趨勢要看週線。日線超賣反彈 + 週線空頭 = 假反彈。

**實作：**
- 日線 resample 成週線（`W-FRI` 週五收盤）
- 計算週線指標，再 forward fill 映射回日線

**新增特徵（2 維）：**

| 特徵 | 說明 |
|------|------|
| `weekly_ma_trend` | 週線收盤 vs 週 MA5（1=多頭排列, -1=空頭排列） |
| `weekly_rsi` | 週線 RSI（0~1 標準化） |

---

#### 4. 市場狀態辨識（+1~2% 預估提升）

**原理：** 多頭、空頭、盤整三種市場的走勢邏輯完全不同，同一套規則不適用所有市場。

**判斷邏輯：**
```
Close > MA20 > MA60 → 多頭（1）：拉回就是買點
Close < MA20 < MA60 → 空頭（-1）：反彈就是逃命點
其他 → 盤整（0）：方向不明
```

**新增特徵（1 維）：**

| 特徵 | 說明 |
|------|------|
| `market_regime` | 1=多頭, 0=盤整, -1=空頭 |

**2884 實測：** 多頭 369 天、空頭 174 天、盤整 451 天（符合金融股走勢特性）

### Bug 修復：預測記錄回填方向錯誤

**問題：** 預測記錄的「實際」「漲跌%」「正確」三個欄位從未被成功回填。

**根本原因：** `data/prediction_logger.py` 第 125 行，取預測當天收盤價時使用了錯誤的搜尋方向：
```python
# 舊（錯誤）：往「之前」找，若 pred_date 是假日會拿到更早的錯誤價格
pred_close = _near_price(price_map, pred_date, "before")

# 新（正確）：往「之後」找，確保拿到預測當天或最近交易日的收盤價
pred_close = _near_price(price_map, pred_date, "after")
```

**額外修復：** 新增 `target_date > today` 提早跳過判斷，避免目標日期還沒到就嘗試回填。

### Bug 修復：matplotlib 中文字型方塊亂碼

**問題：** 趨勢圖中所有中文字（標題、軸標籤、圖例、提示文字）全部顯示為方塊 `□□□□`。

**根本原因：** matplotlib 預設使用 `DejaVu Sans` 字型，不包含 CJK 字元。原本 `_cjk_font()` 只套用在標題，其他文字全部漏掉。

**修復：** 在 `accuracy_trend_dialog.py` 的 `_build_canvas()` 開頭全局設定中文字型：
```python
plt.rcParams["font.sans-serif"] = ["Microsoft JhengHei", "SimHei", "Arial"]
plt.rcParams["axes.unicode_minus"] = False
```

所有 matplotlib 文字一次解決，不需逐一指定 `fontproperties`。

### 打包流程重構：factory_defaults 原廠範本機制

**問題：** 舊的打包流程需要先手動清空開發者自己的 config.json、自選股、模型等資料，再執行打包。這有兩個風險：
1. **誤刪開發者資料** — 打包完忘記補回 API Key、自選股等
2. **未來新增檔案時漏清或誤刪** — 寫死「要刪什麼」的清單，隨功能增加越來越難維護

**解決方案：** 引入 `factory_defaults/` 原廠範本目錄

**核心原則：**

> ★★★ 打包流程「絕對不動」專案原始檔案。只動 dist 資料夾。 ★★★

**機制：**
1. `factory_defaults/` 目錄存放所有需要「乾淨版」的檔案範本
2. PyInstaller 照常打包（用開發者目前的檔案，包含 API Key 等都沒關係）
3. 打包完成後，`build.py` 自動用 `factory_defaults/` 裡的範本**覆蓋 dist 資料夾**中的對應檔案
4. 同時清空 dist 裡的 `models/saved/`、`cache/chip/`、`logs/`
5. 最後壓成 ZIP → 發布版本是乾淨的，開發者的資料完全不受影響

**目前 factory_defaults/ 包含：**

| 檔案 | 內容 |
|------|------|
| `config.json` | API Key 全空、`welcome_shown: false` |
| `watchlist.json` | `{"symbols": []}` |
| `recent.json` | `{"symbols": []}` |
| `prediction_log.csv` | 只有表頭（無任何預測記錄） |

**未來維護規則：**

> ★ 如果新增了「使用者個人資料」相關的設定檔或資料檔，
> 　請同步在 `factory_defaults/` 新增對應的乾淨範本。
>
> ★ 如果 config.json 新增了欄位，
> 　請同步更新 `factory_defaults/config.json`。
>
> ★ 絕對不要在 build.py 中對專案根目錄的檔案做任何刪除或修改操作。
> 　所有清理動作都只針對 dist/ 資料夾。

### Bug 修復：Win10 無法啟動（TensorFlow DLL 載入失敗）

**問題：** 打包後的程式在部分 Win10 電腦閃退，錯誤訊息：
```
DLL load failed while importing _pywrap_tensorflow_lite_metrics_wrapper: 找不到指定的模組。
```

**根本原因：** TensorFlow Lite 的原生模組（`_pywrap_tensorflow_lite_metrics_wrapper.pyd`）執行時依賴 `vcruntime140.dll` 和 `vcruntime140_1.dll`（Visual C++ Redistributable）。Win11 和部分 Win10 電腦已預裝，但乾淨的 Win10 電腦沒有，導致 DLL 載入失敗。

**修復：** 在 `stock_predictor.spec` 的 `binaries` 中加入 VC++ Runtime DLL：
```python
_vc_dir = os.path.join(VENV, 'Lib', 'site-packages', 'PySide6')
vc_binaries = [
    (os.path.join(_vc_dir, 'vcruntime140.dll'),   '.'),
    (os.path.join(_vc_dir, 'vcruntime140_1.dll'),  '.'),
]
```

**DLL 來源：** PySide6 套件自帶的版本（與打包環境一致），不從系統目錄取用。

> ★ 注意：這兩個 DLL 必須放在 dist 根目錄（`.`），與 exe 同層。
> 　放在子目錄裡 Windows 的 DLL 搜尋路徑找不到，一樣會閃退。
