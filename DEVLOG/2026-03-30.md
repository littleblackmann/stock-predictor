### 1. 自動更新版本號未更新 — 根因分析與修復（v1.3.1）

#### 根因
Day 9 打包 v1.3.0 時打了**兩次**（第一次忘記更新 changelog → 刪除 Release → 改好 → 第二次打包）。`build_manifest.json` 在第一次打包時被覆寫成 v1.3.0 的 manifest，導致第二次打包的 patch 比對基準是「第一次 v1.3.0 vs 第二次 v1.3.0」，只有 3 個檔案差異，**不是 v1.2.8→v1.3.0 的完整差異**。

結果：
- patch zip 遺漏了 `_internal/version.json`（兩次 v1.3.0 hash 相同）
- 爸的 Win10 電腦更新後 exe 換了（changelog 有更新），但 version.json 還是 1.2.8
- 版本號卡住 → 每次啟動都重複跳更新通知（無限循環）

#### 修復（3 道防線）

**防線 1：build.py — 版本化 manifest 檔名**
- `build_manifest.json` → `build_manifest_v{版本}.json`
- 新增 `_find_previous_manifest()` 函式：自動找上一個版本的 manifest 作基準
- 重複打包同版本只覆寫自己的 manifest，不影響前一版基準線
- 相容舊的 `build_manifest.json`（legacy fallback）

**防線 2：build.py — 強制包含 version.json**
- `create_patch_zip()` 中強制加入 `_internal/version.json`
- 即使 hash 沒變也會被放進 patch

**防線 3：auto_updater.py — 取最高版本**
- `get_current_version()` 從「讀到第一個就回傳」改為「讀取所有來源，回傳最高版本號」
- 來源：`_MEIPASS/version.json`（xcopy 更新）+ `APP_ROOT/version.json`（bat 寫入）
- 任一來源有正確版本就不會卡住

**影響範圍**：`build.py`、`updater/auto_updater.py`

#### 發佈 v1.3.1
- 僅上傳完整包（不上傳 patch），因為沒有正確的 v1.2.8 manifest 可比對
- 爸的電腦 fallback 到完整包下載（644 MB），一次覆蓋所有檔案
- **實測成功**：爸的 Win10 電腦更新後版本正確顯示 v1.3.1 ✅

### 5. QFont::setPointSize 警告修復

#### 問題
打開設定視窗的「API 設定」分頁時，console 會跳出 5 次 `QFont::setPointSize: Point size <= 0 (-1)` 警告。

#### 根因
`QComboBox`（AI 模型選擇下拉框）沒有設定 stylesheet，裡面有 separator 和 disabled header items。Qt 在計算 combo box 的 size hint 時，需要測量每個 item 的 font metrics，而 separator / disabled items 的 font point size 預設為 -1（未設定），觸發警告。

#### 修復
為 QComboBox 加上明確的 `font-size: 13px` stylesheet（含 `QComboBox QAbstractItemView`），讓 Qt 在計算 font metrics 時有明確的字型大小。

**影響範圍**：`ui/settings_dialog.py`

### 7. Transformer 取代 LSTM — 模型架構大升級

#### 背景
LSTM 回看窗口只有 60 天，無法捕捉台股的季節性規律（除權息、財報季、年底作帳等跨月/跨季模式）。Transformer 的 Self-Attention 機制可以同時看整段序列的任何位置，不會有 LSTM 的「遺忘」問題。

#### 架構變更

```
舊：股價(60天, 17維) → LSTM(128→64) → 64維 ─┐
                                               ├→ LightGBM → 預測
                         36維原始特徵 ──────────┘

新：股價(300天, 28~41維) → Transformer(3層) → 64維 ─┐
                                                      ├→ LightGBM → 預測
                              36維原始特徵 ────────────┘
```

#### Transformer Extractor 架構
```
Input (300, n_features)
  → Linear Projection → (300, 64)
  → + Sinusoidal Positional Encoding
  → TransformerEncoder × 3 (Pre-LN)
      每層：LayerNorm → MultiHeadAttention(4 heads) → FFN(128, GELU) → Residual
  → Global Average Pooling → (64,)
  → Dense(64, ReLU)  ← feature_vector（萃取層）
  → Dense(1, sigmoid) ← 輔助訓練用
```

- **總參數**：107,329（419 KB，比 LSTM 的 ~80K 稍多但仍很輕量）
- **輸出**：64 維特徵向量（與 LSTM 相同，LightGBM 無需改介面）

#### Transformer 輸入擴充（17 → 28~41 維）
LSTM 只吃 OHLCV + 基礎技術面 + 美股隔夜（17 維），Transformer 可處理更高維度：
- OHLCV（5 維）
- 完整技術面（13 維，含 MACD signal、vol_change 等）
- 美股完整隔夜訊號（4 維，含 VIX）
- 多時間框架（2 維，週線動量）
- 市場行情狀態（4 維）
- 籌碼面衍生特徵（13 維，若有）

新增 `feature_engineer.get_transformer_input_cols()` 方法。

#### 資料量擴充
- yfinance 請求天數 **1500 → 2500**（約 7 年，~1650 筆有效 K 線）
- 美股隔夜資料同步擴充至 2500 天
- 訓練樣本：1650 - 300 = ~1350 筆（舊版只有 ~940 筆）

#### 時間衰減權重（新增）
訓練時依資料日期分配 `sample_weight`：
- 最近 2 年 → 1.0（最重要）
- 2~4 年前 → 0.7
- 4~6 年前 → 0.4
- 6+ 年前 → 0.2

#### 訓練配置
| 參數 | LSTM（舊） | Transformer（新） |
|------|-----------|-------------------|
| 回看窗口 | 60 天 | 300 天 |
| 輸入維度 | 17 | 28~41 |
| Epochs | 30 | 40 |
| 學習率 | 0.001 | 5e-4（Adam） |
| EarlyStopping | patience=5 | patience=8 |
| Batch Size | 32 | 32 |
| Dropout | 0.2 | 0.15 |
| 特殊機制 | — | 時間衰減權重、Positional Encoding |

#### 批次特徵萃取（記憶體安全）
Transformer 窗口 300 天遠大於 LSTM 的 60 天，全部一次建構會吃太多記憶體。
改為分批 64 筆建構 + 推論，避免 OOM。

#### LightGBM 搭配重訓機制
Transformer 和 LSTM 的 64 維特徵語義完全不同，但維度相同（都是 64），dimension check 會誤判為相容。
加了防呆：`prediction_worker.py` 中如果 Transformer 是剛訓練的（`seq_loaded=False`）但 LightGBM 載入了舊模型 → 強制重訓 LightGBM。

#### 舊模型自動清理（使用者無感遷移）
- 新增 `data_paths.cleanup_legacy_models()` — 啟動時自動清除 `lstm_*` 和 `lgbm_*` 舊模型
- 寫入 `.transformer_migrated` 標記，只清一次
- 使用者更新後第一次預測會自動重訓，完全無感

#### 影響範圍
- **新增**：`models/transformer_extractor.py`
- **修改**：`workers/prediction_worker.py`（LSTM→Transformer、1500→2500、批次萃取、LightGBM 重訓防呆）
- **修改**：`features/feature_engineer.py`（新增 `get_transformer_input_cols()`）
- **修改**：`ui/prediction_progress_dialog.py`（步驟文字 LSTM→Transformer）

### 9. K 線圖日期缺少當天資料（v1.4.1）

#### 問題
K 線圖最後一根 K 棒日期停在 2026-03-27（週五），但今天是 2026-03-30（週一），22:50 已收盤超過 9 小時，當天資料應該要有。

#### 根因
`yfinance_adapter.py` 的 `fetch_history()` 使用：
```python
end_date = datetime.today()  # 2026-03-30
end=end_date.strftime("%Y-%m-%d")  # "2026-03-30"
```
yfinance 的 `end` 參數是 **exclusive**（不含當天），所以 `end="2026-03-30"` 只會拿到 2026-03-29 及之前的資料。最後一個交易日是 2026-03-27（週五）。

#### 修復
```python
end_date = datetime.today() + timedelta(days=1)  # +1 讓 yfinance 包含今天
```

**影響範圍**：`data/yfinance_adapter.py`

### 10. 啟動偶發閃退（v1.4.1）

#### 問題
有時候剛打開系統就直接被關掉，沒有任何錯誤訊息。

#### 根因分析（3 個漏洞）

**漏洞 1：啟動序列無保護**
`main.py` 的 `migrate_from_old_location()` 和 `cleanup_legacy_models()` 在 QApplication 建立前執行，沒有 try/except。如果檔案被鎖（防毒軟體 / Windows Search 索引 / exe 未關閉），`os.remove()` 或 `shutil.copy2()` 拋出 `PermissionError`，app 在建立任何視窗前就 crash，完全沒有 log（logging 還沒初始化）。

**漏洞 2：AppLoader done signal 未發射**
`AppLoader.run()` 的 exception handler 只 log 錯誤但不 emit `done` signal。主視窗建立在 `_on_load_done()` callback 中，沒有 `done` signal 就永遠不會觸發 → splash screen 永遠卡住，使用者看起來像「開了又關了」。

**漏洞 3：模型清理單檔失敗連鎖**
`cleanup_legacy_models()` 中 `os.remove()` 沒有 try/except，一個檔案鎖定就整個函式 crash。

#### 修復（3 處）

| 位置 | 修復 |
|------|------|
| `main.py:128-130` | `migrate_from_old_location()` + `cleanup_legacy_models()` 包在 try/except 中，失敗不阻擋啟動 |
| `main.py:121` | AppLoader exception handler 補上 `self.sig.done.emit()`，確保主視窗至少會出現 |
| `data/data_paths.py:134,138` | 每個 `os.remove()` 獨立 try/except，單檔鎖定不影響其他清理 |

**影響範圍**：`main.py`、`data/data_paths.py`
